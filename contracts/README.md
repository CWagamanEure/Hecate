# Hecate on-chain verifier (v1 stub)

`HecateSettlementVerifier.sol` is a minimal Solidity adapter that proves a
Hecate batch-receipt engine signature on-chain. It does **not** reproduce
canonical JSON, matching, settlement, or conservation invariants. It
proves: the supplied 32-byte body hash was signed by the supplied engine
address.

This is consistent with the v1 signing format (see
`shared/crypto/signing.ts`): raw `keccak256(canonicalJson(body))` with no
EIP-191 prefix and no EIP-712 domain. Callers compute the body hash off
chain (via the same canonical-JSON serializer the engine uses) and pass
the resulting `bytes32` plus the 65-byte signature to the contract.

Full on-chain re-verification (recomputing the body hash from typed
fields, plus settlement / conservation / receipt-binding invariants) is
roadmap material — see [docs/ROADMAP.md §4](../docs/ROADMAP.md) and §5
(EIP-712 migration).

## Running the Forge test

Requires [Foundry](https://book.getfoundry.sh/) (`foundryup`).

```sh
cd contracts
forge install foundry-rs/forge-std    # first time only
forge test -vv
```

Expected: **9 tests pass**, including
- `test_verifyHonestReceipt`
- `test_recoverSignerReturnsEngine`
- `test_rejectsWrongEngine`
- `test_rejectsTamperedHash`
- `test_rejectsTamperedSignature`
- `test_rejectsBadLengthSignature`
- `test_rejectsZeroExpectedEngine`
- `test_verifyAndEmitEmitsEvent`
- `test_verifyAndEmitNoEventOnFailure`

The fixture `(HASH, SIG, ENGINE)` in `test/HecateSettlementVerifier.t.sol`
is generated by `scripts/gen-onchain-fixture.ts`. It is deterministic:
canonical JSON is deterministic by spec, and `@noble/secp256k1` uses
RFC 6979 deterministic ECDSA.

## Regenerating the fixture

If the canonical-JSON serializer or the receipt schema changes, the
hard-coded hash in the test file becomes stale. Regenerate:

```sh
npx tsx scripts/gen-onchain-fixture.ts
```

Outputs three Solidity-pasteable constants. Replace the corresponding
lines in `contracts/test/HecateSettlementVerifier.t.sol`.

To verify a real saved bundle instead of the contrived body:

```sh
npx tsx scripts/gen-onchain-fixture.ts --bundle ./data/last-bundle.json
```

## Deploying

### Local anvil (development; no real funds)

```sh
# In terminal 1: start a local node.
anvil

# In terminal 2: deploy. anvil's first default key is well-known and
# pre-funded with 10000 ETH on the local chain.
cd contracts
forge script script/Deploy.s.sol \
  --rpc-url http://localhost:8545 \
  --broadcast \
  --private-key 0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80

# Note the deployed address from the script output.

# Verify a real Hecate bundle on the local deployment:
cd ..
SEPOLIA_RPC_URL=http://127.0.0.1:8545 \
VERIFIER_ADDRESS=<deployed-address> \
DEPLOYER_PRIVATE_KEY=0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80 \
  npm run onchain:verify -- ./data/last-bundle.json
```

### Sepolia (production demo)

Prerequisites — fill these into `.env` in the project root (see
[`.env.example`](../.env.example) for the full block):

- `ALCHEMY_API_KEY` — bare API key from https://dashboard.alchemy.com/
  (scripts auto-build the Sepolia URL). Alternatively `SEPOLIA_RPC_URL`
  for any other provider.
- `DEPLOYER_PRIVATE_KEY` — fresh wallet, ~0.005 Sepolia ETH funded
  ([Alchemy faucet](https://www.alchemy.com/faucets/ethereum-sepolia)).
- `ETHERSCAN_API_KEY` — free from https://etherscan.io/myapikey
  (optional but recommended; enables `forge --verify` source upload).

Once `.env` is set:

```sh
cd contracts
# Forge auto-loads .env from the project root for ${VAR} substitution in
# foundry.toml's [rpc_endpoints]. The named endpoint `sepolia-alchemy` is
# defined as https://eth-sepolia.g.alchemy.com/v2/${ALCHEMY_API_KEY}.
forge script script/Deploy.s.sol \
  --rpc-url sepolia-alchemy \
  --broadcast \
  --verify \
  --etherscan-api-key $ETHERSCAN_API_KEY \
  --private-key $DEPLOYER_PRIVATE_KEY

# If you set SEPOLIA_RPC_URL instead of ALCHEMY_API_KEY, use --rpc-url sepolia.
```

Record the deployed address (printed at end of script) and paste it into
`.env`'s `VERIFIER_ADDRESS=` line. Then run on-chain verification of a real
Hecate bundle:

```sh
cd ..
npm run dev &  # terminal 1, engine
npm run simulate -- --reset-demo-state --data-dir ./data --save-bundle ./data/last-bundle.json
npm run onchain:verify -- ./data/last-bundle.json
```

`npm run onchain:verify` reads `.env` automatically via the project's
`loadDotenv` helper.

The script prints the Sepolia tx hash plus a `https://sepolia.etherscan.io/tx/...`
link. Following the link shows the `ReceiptVerified(bytes32 indexed hash,
address indexed signer)` event with the bundle's body hash and the engine
address.

`--dry-run` simulates via `eth_call` only (no tx, no gas spent):

```sh
npm run onchain:verify -- ./data/last-bundle.json --dry-run
```

The script enforces chain-id safety: it refuses to broadcast on any chain
other than Sepolia (11155111) or local Anvil (31337).

## v1 caveats

- **Signature scheme is v1-local, not wallet UX.** The engine signs a
  canonical-JSON hash directly. EIP-712 migration (which would let the
  contract recompute the hash from typed-data fields on chain) is the
  right post-v1 path — see the TODO note in `shared/crypto/signing.ts`.
- **No on-chain conservation check.** v1 produces receipts whose
  conservation invariants are checked off chain by `verifyFullBatch`. A
  future revision could compute them in Solidity.
- **No on-chain settlement re-verification.** Likewise off-chain only.
- **Local engine key under LOCAL_MOCK and EIGEN_TEE.** The engine signing
  key is the same `ENGINE_PRIVATE_KEY` env var in both modes today
  (`signer.mode = LOCAL_DEV_KEY` in `/attestation`). Real Eigen
  app-wallet signing is future work.
